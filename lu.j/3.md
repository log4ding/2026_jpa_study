# 3장. 영속성 관리

## 3.1 엔티티 매니저 팩토리와 엔티티 매니저

- 일반적으로 엔티티 매니저 팩토리는 1개만 만든다.
- 엔티티 매니저 팩토리에서 앤티티 매니저를 생성한다.

```java
EntityManager em = emf.createEntityManager();
```

- 엔티티 매니저 팩토리는 Thread safe 하므로 다른 스레드 간에 공유가 가능하지만 엔티티 매니저는 여러 스레드가 동시에 접근할 경우 동시성 문제가 발생하므로 스레드 간 공유를 하지 않아야 한다.
- 엔티티 매니저는 데이터베이스 연결이 꼭 필요한 시점까지 커넥션을 얻지 않는다.

## 3.2 영속성 컨텍스트란?

- 엔티티를 영구 저장하는 환경 이라는 뜻
- 엔티티 매니저로 엔티티를 저장하거나 조회할 경우 엔티티 매니저는 영속성 컨텍스트에 엔티티를 보관하고 관리한다.

```java
em.persist(member);
```

- 위 커드는 엔티티 매니저를 사용해 회원 엔티티를 영속성 컨텍스트에 저장한다.
- 여러 엔티티 매니저가 같은 영속성 컨텍스트에 접근할 수도 있다.

## 3.3 엔티티의 생명주기

- 비영속(new/transit): 영속성 컨텍스트와 전혀 관계가 없는 상태
  - 엔티티 객체를 생성한 상태
  ```java
  Member member = new Member();
  member.setId("member1");
  member.setUsername("회원");
  ```
- 영속(managed): 영속성 컨텍스트에 저장된 상태
  - 영속성 컨텍스트가 관리하는 엔티티를 영속 상태라고 한다.
  ```java
  em.persist(member);
  ```
- 준영속(detached): 영속성 컨텍스트에 저장되었다가 분리된 상태
  - 영속성 컨텍스트가 관리하던 영속 상태의 엔티티를 영속성 컨텍스트가 관리하지 않으면 준영속 상태가 된다.
  - 준영속 상태로 만들려면 `em.detach()` 를 호출
- 삭제(removed): 삭제된 상태

  - 엔티티를 DB와 영속성 컨텍스트에서 삭제한다.

  ```java
  em.remove(member);
  ```

  ## 3.4 영속성 컨텍스트의 특징

- 영속성 컨텍스트와 식별자 값
  - 영속 상태는 식별자 값이 반드시 있어야 한다. (@Id)
- 영속성 컨텍스트와 데이터베이스 저장
  - JPA 는 트랜잭션을 커밋하는 순간 영속성 컨텍스트에 새로 저장된 엔티티를 데이터베이스에 반영한다. 이를 flush 라고 한다.
- 영속성 컨텍스트가 엔티티를 관리하면 좋은 장점
  - 1차 캐시
  - 동일성 보장
  - 트랜잭션을 지원하는 쓰기 지연
  - 변경 감지
  - 지연 로딩

### 3.4.1 엔티티 조회

- 영속성 컨텍스트는 내부에 캐시를 가지고 있는데 이를 1차 캐시 라고 한다.
- 영속성 컨텍스트 내부에 Map 이 있는데 키는 @Id 로 매핑한 식별자고 값은 엔티티 인스턴스

```java
Member member = new Member();
member.setId("member1");
member.setUsername("회원1");

em.persis(member);
```

- 위 코드를 실행하면 1차 캐시에 회원 엔티티를 저장, 데이터베이스엔 저장되지 않는다.

```java
Member member = em.find(Member.class, "member1");
```

- em.find() 를 호출할 경우 1차 캐시에서 찾고 없다면 엔티티에서 조회한다.
- 이를 통해 성능상 이점을 누릴 수 있다.
- 영속 엔티티의 동일성 보장도 가능하다.

- _(개인적인 의견)_

  - 엔티티 매니저 단위의 메모리 캐시라서 범위가 작아서 실무에 크게 장점이 있을까? redis cache 를 쓸 수 밖에 없지 않을까. 라는 생각
  - 한 요청 안에서 여러번의 같은 엔티티를 호출하는 특정 상황에서는 장점이 있을 순 있음

### 3.4.2 엔티티 등록

```java
EntityManager em = emf.createEntityManager();
EntityTransaction transaction = em.getTransaction();
transaction.begin(); // 트랜잭션 시작

em.persist(memberA);
em.persist(memberB);
// 여기까지 INSERT SQL 을 데이터베이스에 보내지 않음

transaction.commit(); // 트랜잭션 커밋
```

- 엔티티 매니저는 트랜잭션을 커밋하기 직전까지 엔티티를 저장하지 않고 내부 쿼리 저장소에 모아둔다.
- 트랜잭션 커밋할 때 모아둔 쿼리를 데이터베이스에 보내는데 이를 *쓰기 지연*이라고 한다.

- _(개인적인 의견)_
  - 네트워크 비용 감소로 인해 (한번에 SQL 을 보내기 때문에) 성능상의 이점은 있지 않을까?
  - 트랜잭션의 단위가 커질 경우 메모리에 모아두는 쿼리들이 많아지게 되므로 한 단위 조정이 필수일 것으로 보임

### 3.4.3 엔티티 수정

#### SQL 수정 쿼리의 문제점

- 요구사항이 많아질수록 수정 쿼리가 많아지는 것은 물론이고 비즈니스 로직을 분석하기 위해 SQL 을 계속 확인해야함
- 비즈니스 로직이 SQL 에 의존적이게 된다.

#### 변경감지

- JPA 는 엔티티를 어떻게 수정할까?

```java
EntityManager em = emf.createEntityManager();
EntityTransaction transaction = em.getTransaction();
transaction.begin();

Member memberA = em.find(Member.class, "memberA");

// 영속 엔티티 데이터 수정
memberA.setUsername("hi");
memberA.setAge(10);

// em.update(member) // 이런 코드가 필요하지 않을까?

transaction.commit();
```

- JPA 는 엔티티의 변경을 자동으로 감지하게 되는데 이를 dirty checking (변경감지)라고 한다.
- 변경감지는 영속성 컨텍스트가 관리하는 영속 상태의 엔티티에만 적용된다.
- 그리고 JPA 는 변경된 엔티티의 컬럼만 업데이트 하지 않고 모든 필드를 업데이트 하게 된다.

  - 장점
    - 모든 필드를 사용하면 수정 쿼리가 항상 같음
    - 데이터베이스에 동일한 쿼리를 보냄으로써 DB는 한번 파싱된 쿼리를 재사용 가능

- _(개인적인 의견)_
  - @DynamicUpdate, @DynamicInsert 를 통해 변경된 필드만 업데이트 할 수 있도록 하는 것이 더 나을거 같다.
  - 인덱스에 걸린 컬럼들을 변경하지 않았는데 모든 컬럼 변경으로 쓰기 비용이 증가할 수도 있을 것 같음
    - 좀더 알아보니 같은 값을 업데이트 할 경우 인덱스 갱신이 꼭 되는가는 DB에 따라 다를 수 있다고 함. 그래도 어노테이션을 사용하는 것이 더 나은 선택이라고 보여짐

### 3.4.4 엔티티 삭제

```java
em.remove(member);
```

- remove 가 호출되는 순간 영속성 컨텍스트에서 제거 된다.
- 이렇게 제거된 엔티티는 재사용하지 말고 GC의 대상이 되도록 놔두는 것이 낫다.

## 3.5 플러시

- 플러시는 영속성 컨텍스트의 변경 내용을 데이터베이스에 반영한다.

### 직접 호출

- 엔티티 매니저의 flush() 를 직접 호출
- 테스트나 다른 프레임워크와 JPA 를 함께 사용할 때 제외하고 거의 사용하지 않음

### 트랜잭션 커밋 시 플러시 자동 호출

- 데이터베이스에 변경 내용을 SQL 로 전달하지 않고 트랜잭션만 커밋하면 반영되지 않는다.
- 그래서 JPA 는 트랜잭션을 커밋할 떄 flush 를 자동으로 호출하게 되어 있다.

### JPQL 쿼리 실행 시 플러시 자동 호출

- JPQL, Critetia 같은 객체지향 쿼리를 호출할 때도 플러시가 실행된다.

```java
em.persist(memberA);
em.persist(memberB);
em.persist(memberC);

// 중간에 JPQL 실행
query = em.createQuery("select m from Member m", Member.class);
List<Member> members = query.getResultList();
```

- memberA, memberB, memberC 가 아직 데이터베이스에 반영되지 않은 상태에서 JPQL은 DB에서 엔티티를 조회한다.
- 이런 케이스를 막기 위해 JPA 는 JPQL 을 사용할 때 자동으로 플러시를 호출한다.
- find() 할 때는 호출하지 않는다.

### 3.5.1 플러스 모드 옵션

- FlushModeType.AUTO: 커밋이나 쿼리를 실행할 때 플러시 (기본)
- FlushModeType.COMMIT: 커밋할 때만 플러시
- 플러시라는 이름으로 인해 영속성 컨텍스트에 보관된 엔티티를 지운다고 생각하지 않도록 한다.
- 플러시는 영속성 컨텍스트의 변경 내용을 데이터베이스에 동기화 하는 것이다.

## 3.6 준영속

- 영속 상태의 엔티티가 영속성 컨텍스트에서 분리된 것을 준영속 상태라고 한다.

- _(개인적인 의견)_
  - 이전에 안되었던 내용을 다시 훑어보면,

```java
Member member = new Member("alice", 20);
member = repository.save(member);
member.setAge(40);

Member member2 = repository.find("alice");

// member2 의 age 가 40으로 나옴
```

### 3.6.1 엔티티를 준영속 상태로 전환: detach()

```java
Member member = new Member();
member.setId("id1");
member.setUsername("멤버1");
em.persist(member);

em.detach(member);
transation.commit();
```

- 이미 준영속 상태기 때문에 영속성 컨텍스트가 지원하는 어떤 기능도 동작하지 않음
- 영속성 컨텍스트로부터 분리된 상태다.

### 3.6.2 영속성 컨텍스트 초기화: clear()

- clear() 의 경우 영속성 컨텍스트의 모든 엔티티를 준영속 상태로 만든다.

- _(개인적인 의견)_
  - 어떤 상황에서 해당 함수가 쓰일 수 있을지 ?
    - 대량 배치 작업에서 영속성 컨텍스트에 많은 엔티티가 쌓였을 경우 한번 clear() 를 해주고 다음 대량 배치 작업 진행??

```java
Member member = em.find(Member.class, "memberA");
em.clear();
member.setUsername("changeName");
```

### 3.6.3 영속성 컨텍스트 종료: close()

- 영속성 컨텍스트를 종료하면 해당 영속성 컨텍스트가 관리하던 영속 상태의 엔티티가 모두 준영속 상태가 된다.
- _(개인적인 의견)_ 보통 spring 에서는 close() 를 호출해주기 때문에 쓸 일은 없지 않을까.

### 3.6.4 준영속 상태의 특징

#### 거의 비영속 상태에 가깝다.

- 영속성 컨텍스트가 제공하는 어떠한 기능도 동작하지 않음

#### 식별자 값을 가지고 있다.

- 비영속 상태는 식별자가 없을 수 있는데 준영속 상태는 식별자가 있다.

#### 지연로딩이 불가능

- 지연 로딩은 프록시 객체를 로딩해두고 해당 객체를 실제로 사용할 때 영속성 컨텍스트를 통해 데이터를 불러옴, 준영속성 상태일 경우 지연 로딩시에 문제가 발생한다.

- _(개인적인 의견)_

  - 준영속 상태가 필요한 이유가 뭘까??

    - 트랜잭션에서 조회한 엔티티를 컨트롤러로 넘김 → 트랜잭션 종료 후 그 엔티티는 준영속
    - 사용자가 화면에서 수정한 값을 다시 서버로 보내면 그 객체는 준영속, 그래서 merge로 다시 붙임

  - 비영속과 준영속의 차이
    - 비영속: DB에 아직 없을 수도 있는 “새 객체” (id 없음/미확정)
    - 준영속: DB에 이미 있는 객체인데, 현재 컨텍스트에서만 관리하지 않는 상태

### 3.6.5 병합: merge()

- merge() 는 준영속 상태의 엔티티를 다시 영속 상태로 변경한다. 준영속 상태의 엔티티를 받아 새로운 영속 상태의 엔티티를 반환

- _(개인적인 의견)_

  - repository 의 save 는 어떻게 동작할까?

  ```java
  @Override
  @Transactional
  public <S extends T> S save(S entity) {

  	Assert.notNull(entity, ENTITY_MUST_NOT_BE_NULL);

  	if (entityInformation.isNew(entity)) {
  		entityManager.persist(entity);
  		return entity;
  	} else {
  		return entityManager.merge(entity);
  	}
  }
  ```

  - 비영속 이었다면 -> persist()
  - 준영속이었다면 -> merge()

#### 준영속 병합

```java
Member member = createMember("memberA", "회원1");
member.setUsername("회원명 변경");  // 준영속 상태에서 변경
mergeMember(member);

static Member createMember(String id, String username) {
  // 영속성 컨텍스트 1 시작
  EntityManager em1 = emf.createEntityManager();
  EntityTransaction tx1 = em1.getTransaction();
  tx1.begin();

  Member member = new Member();
  member.setId(id);
  member.setUsername(username);

  em1.persist(member)

  tx1.commit();
  em1.close();
  // 영속성 컨텍스트 1 종료

  return member;
}

static void mergeMember(Member member) {
  // 영속성 컨텍스트 2 시작
  EntityManager em2 = emf.createEntityManager();
  EntityTransaction tx2 = em2.getTransaction();
  tx2.begin();
  Member mergeMember = em2.merge(member);
  tx2.commit();

  // 준영속 상태
  System.out.println("member = " + member.getUsername());

  // 영속 상태
  System.out.println("mergeMember = " + mergeMember.getUsername());

  em2.close();

  // 영속성 컨텍스트 2 종료
}
```

- merge 는 파라미터로 넘어온 준영속 엔티티를 사용해서 새롭게 병합된 영속 상태의 엔티티를 반환한다.
- 파라미터로 넘어온 엔티티는 병합 후에도 준영속 상태로 남아있다.

#### 비영속 병합

```java
Member member = new Member();
Member newMember = em.merge(member);
tx.commit();
```

- 병합은 파라미터로 넘어온 엔티티의 식별자로 영속성 컨텍스트를 조회 > 없으면 DB 조회 > 여기서도 없으면 엔티티 생성 후 병합
